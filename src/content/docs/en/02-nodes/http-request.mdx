If n8n is a city, the **HTTP Request** node is its highway on-ramp — any service that exposes an API (even if there's no official "app node" for it) can be reached from your workflow through this node. It's the first gate from beginner to "power user", and we're opening it now.

## What it does

In one line: **send an HTTP request, get a response, hand the response to the next node.** Sounds humble — but 90% of "how do I integrate XYZ service" ends up at HTTP Request.

It has just three core parameters:

1. **Method** — GET / POST / PUT / DELETE / PATCH
2. **URL** — supports expression interpolation, e.g. `https://api.example.com/users/{{ $json.id }}`
3. **Authentication** — None / Basic / Header / OAuth2 / Predefined credential

Master these three; the rest is detail.

## Three common authentication patterns

| Scenario | Authentication | Example |
|---|---|---|
| Public API (e.g. GitHub public data) | None | The example above |
| API Key in header | Header Auth | `Authorization: Bearer xxx` |
| Third-party OAuth2 | OAuth2 API / Predefined | Google, Twitter, Notion, etc. |

<TipCallout type="quest" title="Extension challenge">
  Modify the workflow: **only post to Slack when followers count actually changes**. Hint — you need an IF node, plus a way to remember the previous followers count (n8n Static Data, Airtable, or a simple file). We'll cover this pattern in Part 4.
</TipCallout>

## Section recap

- HTTP Request is n8n's universal node — three core params: Method / URL / Authentication
- Switch to expression mode whenever variables appear in the URL
- Authentication trio — None / Header / OAuth2 — covers 95% of cases
- Classic workflow pattern: **Trigger → Fetch → Transform → Notify**

Next up: the [Set node](/en/02-nodes/) — HTTP Request usually returns raw data that needs "reshaping" before downstream nodes can use it.
